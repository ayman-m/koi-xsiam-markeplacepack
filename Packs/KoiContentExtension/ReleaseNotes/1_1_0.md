
#### Playbooks

##### KOI Ext - Hunt Sweep

- **New: KOI Ext - Hunt Sweep** — a **Job-attachable, time-triggered** hunt-sweep playbook. It is triggered by a **Cortex Job on a schedule, NOT by a correlation rule** — the deliberate design for this pack, where KOI is run-on-demand and a scheduled Job is the scan/hunt scheduler. Like *KOI Ext - Unified Script Runner* it closes its own investigation at the end of every run.
- **Runs a configurable set of validated, cheap, high-signal hunts** in parallel via `xdr-xql-generic-query`, each body embedded **verbatim** and parameterised only by `xql_time_frame`:
  - **H2.1** — items carrying compromise-grade KOI findings (MaliciousActivityDetected, Spyware, Ransomware, Exfils\*, Typosquatting, PromptInjectionDetected, …), deduped on `notification_event_id` (the Alerts stream is re-sent ~245×/24h). This is the hunt that returns **real hits** on the tenant (ModHeader, postmark-mcp, lodahs, Chrome Audio Capture).
  - **H2.6** — critical/high KOI known-bad that is **NOT under governance** (no blocklist, no remediation).
  - **H1.3** — install **bursts** (one item across many hosts in a tight window).
  - **H4.2** — **shadow agentic software** (an MCP server or AI agent) executing in `xdr_data` on a KOI-covered host that KOI never inventoried.
- **The hunt SET is an input** (`hunt_set`, default `H2.1,H2.6,H1.3,H4.2`): an operator tunes which hunts run — a subset, or a List value (`${lists.<name>}`) — **without editing the playbook**.
- **Normalises every match into a single `KoiHunt.Matches` array** ({ hunt_id, match_kind, known_bad, item_id, item_name, marketplace, hostname, risk, reason, org_risk_level, exposure, governed, recommendation }). The raw **short-form** event marketplace (`software_windows`, `chrome`, `vsc`) is mapped to the **API long form** (`windows`, `chrome_web_store`, `vscode`) with the same table the parsing rule and *KOI Ext - Investigate Item* use, before it reaches any `koi-` command; `built_in` / `side_loaded` / `ollama` have no API value and are left empty (resolved from inventory instead).
- **Gates on matches:** zero matches → posts *"hunt sweep clean"* and closes. Otherwise it investigates each match through the new **KOI Ext - Hunt Match Investigation** sub-playbook — item matches via *KOI Ext - Investigate Item* (org risk, endpoint/blast radius, blocklist/allowlist governance), host/shadow matches via the Cortex-native `core-get-endpoints`, **surfacing (never running)** a `core-script-run` of the KOI deployment script to refresh a stale host.
- **Posts one war-room markdown table** of every hunt match with its investigation verdict (risk, blast radius, governed?, recommended action).
- **Analyst-gated response, never automatic.** A confirmed known-bad, **ungoverned** item (from H2.1/H2.6) that meets `min_risk` is routed to *KOI Ext - Block and Remediate* with **`auto_block=false`**, so the blocklist write requires human approval. **A scheduled hunt never auto-blocks** — this gate is hard-wired and must not be weakened.
- **Inputs (with defaults):** `hunt_set` (the four hunts), `min_risk` (`high`), `xql_time_frame` (`7 days`), `auto_investigate` (`true`), `max_matches_to_investigate` (`25`, a per-hunt fan-out cap wired to the loops; each hunt's XQL `limit` is a hard backstop), `enable_response_gate` (`true`), `instance_name`.
- **The Cortex XDR - XQL Query Engine is effectively REQUIRED for this playbook** (its purpose is running XQL) — but the pack does **not** hard-depend on it. Every XQL task is `continueonerror`; if the engine is absent the playbook posts *"XQL engine unavailable — hunt sweep skipped"* and closes. The `CortexXDR` dependency stays `mandatory: false` in `pack_metadata.json`.

##### KOI Ext - Hunt Match Investigation

- **New: KOI Ext - Hunt Match Investigation** — sub-playbook that takes ONE hunt match and returns one normalized `KoiHunt.Matches` verdict object. It reuses *KOI Ext - Investigate Item* for item enrichment rather than duplicating it, and gathers Cortex host context with `core-get-endpoints` for host/shadow matches. The whole raw hunt row is passed in and its fields are read here by `coalesce`, so the four hunts' differing column names (`item_id` / `item_key` / `object_id`; `alert_host` / `agent_hostname` / `host_list`; `item_risk`) are all normalized in one place with no forEach column desync. Keeps the main sweep playbook readable.
