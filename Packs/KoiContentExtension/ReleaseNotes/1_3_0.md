#### Parsing Rules
##### KoiContentExtension
Added four promoted columns for **Supply Chain Gateway approval requests**, the only gateway
signal that reaches XSIAM. `approval_requests` audit rows carry `action = null`, so the entire
lifecycle lived in the free-text `message` and was unqueryable; these extract it:
`koi_approval_decision` (submitted / rejected / approved), `koi_approval_requester`,
`koi_approval_decider` and `koi_approval_risk`.
Live-validated on tenant `paet`, 2026-07-23, 90 days, 11 approval rows — decision 11/11,
requester 11/11, decider 5/11, risk 6/11, **zero empty strings**, and **zero leakage** into the
other 20,410 audit rows (extensions 17,110 · devices 2,972 · remediation 264 · policies 37 ·
guardrails 12 · notifications 4), because every column is guarded on `type`.
Two live-proven traps are documented at the site: the `[risk: ...]` segment is **optional**, and
anchoring the requester regex on `]` silently returns null on rows that lack it.

#### Playbooks
##### New: KOI Ext - Gateway Approval Triage
Triages a gateway approval request — the exception a user files after the gateway blocked their
install — and produces an evidence-backed recommendation for a human reviewer. It **never
approves anything**: `koi-allowlist-items-add`, `koi-blocklist-items-add` and
`koi-policy-status-update` are deliberately absent, and the playbook stops at a manual gate.

The design inverts the obvious one, for a structural reason: **a blocked item is not in
inventory**, because the block prevented the install. Live proof — `item_display_name=Snake`
returns `total_count 0` on every marketplace while the control returns 65 chrome_web_store
items. So `koi-inventory-item-get`, `koi-inventory-item-endpoints-list` and the pack's own
**KOI Ext - Enrich Item** are unusable here, and there is no catalog/Koidex command anywhere in
the Marketplace pack's 13 commands to substitute. The inventory lookup is therefore used as a
**detection**: absent means the block held; **present means the blocked item is on endpoints
anyway** (grandfathered, group-scoped, or acquired outside gateway scope).
Also reads policy / blocklist / allowlist context to reconstruct *why* it was blocked (the
gateway never tells XSIAM), and runs one optional XQL query for decision history — how many
prior requests, from how many distinct people, and whether a reviewer already rejected it.
Writes its findings to the alert with `setAlert` (no classifier, no mapper).

#### Documentation
##### Detection queries — new Theme G (Supply Chain Gateway)
Added six live-validated gateway queries (`docs/xql/G1..G6.xql`), taking the library to 51
queries across 5 themes:
- **G1** approval pressure — one blocked item, many requesters (live: *Snake*, 8 requests from
  5 distinct people)
- **G2** re-request after rejection (live: *Snake*, 4 rejections / 4 submissions)
- **G3** installs with no marketplace provenance — gateway-invisible (`null` 286, `built_in` 146,
  `side_loaded` 2)
- **G4** system guardrail remediation, separated from custom-policy noise (`system_malware_protection` 17,
  `system_sideloading_protection` 1; note KOI emits `mcp_registry_protection` **without** the `system_` prefix)
- **G5** rejected-then-installed bypass — **0 rows, the passing result**; single-pass tag-and-reduce
  instead of a join
- **G6** gateway coverage gap — **≈3,088 installs outside PAC scope versus ≈230 inside**, i.e. roughly
  nine in ten installs with a known marketplace never traverse the gateway at all

**Read before writing any gateway detection:** the gateway's own Allowed/Blocked verdict log is
**console-only and never reaches XSIAM** (`koi_koi_raw` has no gateway/block/network type;
`_raw_log contains "Blocked"` returns 0 rows). You cannot alert on a block — Theme G detects its
consequences. Enforcement happens at package download (`clients2.googleusercontent.com/crx/blobs/…`),
not at page browse.

**Identity caveat, recorded once here and at every use site:** on endpoints running the PAC + CA
but no KOI agent the gateway cannot attribute the user (`user_id=unknown&requestedBy=unknown`), so
the end user types their own email into the request form. A live row in this tenant reads
`amahmoud@paltoaltonetworks.com` — a typo of the real domain. Treat the requester as a claim, never
as an identity, and never gate an approval on it.
