
#### Parsing Rules

##### KOI Content Extension Parsing Rule

- **New: KOI Content Extension Parsing Rule**, bound to `vendor="koi", product="koi"` and targeting the **koi_koi_raw** dataset with `no_hit=keep`.
- Promotes **31 flat columns** — the item, device, MCP-server and finding attributes that the Marketplace KOI pack delivers only as JSON strings inside *resources*, *observables*, *finding_info* and *metadata* — including *item_id*, *item_type*, *item_marketplace*, *item_risk_level*, *item_package_name*, *item_criticality*, *device_id*, *device_os_obs*, *device_serial*, *device_last_seen*, *device_last_user*, *device_status*, *alert_hostname*, *finding_uid*, *finding_title*, *koi_event_id*, *koi_notification_id*, *koi_product_version* and the *mcp_\** family.
- Extraction is coalesce-by-*.type* rather than by array position, because `resources[0].type` is *item* on most alerts and *mcp* on the rest.
- Sets *_time* on Audit rows from *created_at*, guarded by `source_log_type = "Audit"` so alert rows keep the timestamp the integration already supplies.
- Promotes **two marketplace columns in the Koi API vocabulary** — *item_marketplace_api* (Alerts) and *marketplace_api* (Audit) — alongside the untouched raw event values. Koi emits short display forms (*chrome*, *vsc*, *software_windows*) that the Koi API rejects with **HTTP 400**; only *npm* and *pypi* are spelled the same in both vocabularies. The *_api* columns are what a playbook must pass to the *marketplace* argument of `koi-inventory-item-get`, `koi-inventory-item-endpoints-list` and `koi-blocklist-items-add`.
- *built_in*, *side_loaded* and *ollama* map to **null** in the *_api* columns: the first two are *installation_method* values leaking into the *marketplace* field and the third is absent from the API list, so none of them is a valid command argument.
- **New: *koi_notification_id*, from `metadata.notification_event_id` — the alert-duplication fix.** The Marketplace integration re-sends every still-open alert on every fetch cycle (*eventFetchInterval* defaults to 1 minute), so **koi_koi_raw holds one row per alert per fetch, not one row per alert**. Measured 21 July 2026 on tenant *api-ayman.xdr.eu*: **1,048 alert rows over 90 days resolve to 317 distinct notifications (≈3.3×)**, and **734 rows over the last 24 hours resolve to 3 (≈245×)**. *koi_notification_id* is the only correct dedupe key — it is 100% populated and is a verified 1:1 identity for *(item.id, device.id, finding_info.uid, finding_info.created_time)*. The alternatives are all wrong at different granularities: *finding_uid* is the **policy** id (3 distinct across 1,040 alerts), *koi_event_id* is the **scan batch** id (20 distinct), and the row *_id* is unique per duplicate. **Anything that counts alerts must use `count_distinct(koi_notification_id)`; a plain `count()` is wrong by two orders of magnitude.**
- **New: *koi_product_version*, from `metadata.version` — a change detector, not an analytic dimension.** It was constant (*1.7.0*) on every row at time of check, so it partitions nothing and nothing should group by it. It exists because this pack has already been caught out once by an undated, unannounced Koi-side payload change (see the *koi_tenant_name* regression below); with this column, dating the next one is a single `comp count(), min(_time), max(_time) by koi_product_version` instead of bisecting *_insert_time* by hand.
- **New: an empty-string guard on *alert_item_version*.** MCP alerts send `{"name":"item.version","value":""}` — the key is present, so `x != null` passes on 100% of rows while the value is useless. Measured 21 July 2026, *item.version* was the empty string on **831 of 842** MCP alerts. A separate `alter` stage now rewrites the empty string to a true null, so population audits, "group by version" widgets and playbook branches testing `!= null` stop being wrong on the whole MCP cohort. This is the only field in the pack where the standard null test gives a false pass; every other promoted alert column was re-checked and came back clean.
- Note: parsing rules apply at ingest only. Rows already present in **koi_koi_raw** are not reprocessed, so the promoted columns are null for historical data until new events arrive.

#### Modeling Rules

##### KOI Content Extension Modeling Rule

- **New: KOI Content Extension Modeling Rule**, mapping the **koi_koi_raw** dataset to the Cortex Data Model with a separate block for *Alerts* and *Audit* rows.
- The Alerts block depends on the columns promoted by the KOI Content Extension Parsing Rule, and the Audit block reads *marketplace_api*, which is also a parsing-rule column. **Neither block is standalone; the two rules must be deployed together.** Deployed alone, the modeling rule maps nulls for 12 of the Alerts mappings and for *xdm.target.resource.sub_type* on Audit.
- *xdm.target.resource.sub_type* now carries the **API** marketplace vocabulary on both blocks (*item_marketplace_api* / *marketplace_api*) instead of the raw short event form, so one XDM field holds one vocabulary and its values are directly usable as command arguments.
- **Two mappings were removed after live measurement on 21 July 2026, and must not be re-added:**
  - ***xdm.alert.original_alert_id* (was *finding_uid*).** *finding_uid* is a **policy** identifier, not an alert identifier — 3 distinct values across 1,040 alerts (*20940 "MCP Servers alerts"*, *23300 "NPM Block CS"*, *20907 "yito test"*). Mapping it would have collapsed 1,040 alerts onto 3. *original_alert_id* means "the id the vendor gave this alert", and Koi supplies no such per-alert identifier on this stream: *koi_notification_id* is a notification/delivery id, correct for dedupe but not a vendor alert identity. The field is left unmapped deliberately.
  - ***xdm.target.host.fqdn* (was *alert_hostname*).** Hostnames contain a dot on **0 of 1,138** alert rows — they are bare names (*M-HFQQ44F5XF*, *Gary的MacBook Air*). Mapping a bare name to *fqdn* is worse than leaving it null, because a cross-source join on *fqdn* then fails silently. *xdm.target.host.hostname* already carries the value correctly.
  - An empty field is a true statement; a populated field that is wrong is a silent join failure. Both removal sites carry an inline DO-NOT-RE-ADD comment.
- *xdm.event.id* stays mapped to *_id*. It is an XSIAM-internal column, so it survives without the parsing rule, and *_id* is the only per-row unique value. A dedupe key is not an event identity.
- *xdm.observer.name* and *xdm.observer.unique_identifier* gain a **_collector_name fallback**. Koi has stopped emitting *koi_tenant_name* and *koi_customer_id* — populated on **20,324 of 20,522 rows over 90 days but on 0 of the 75 rows ingested in the last three hours** (measured 21 July 2026; the same regression was reproduced on 19 events this project generated itself). The original mapping is kept for the historical data; *_collector_name* is 100% populated and preserves the same tenant partition (*plts* ↔ *KOI_PLTS*, *paet* ↔ *KOI_PAET*). Re-check whether the Koi-side change is transient before removing the fallback.
- The modeling-rule schema declares *metadata* alongside the other raw JSON payload fields (*finding_info*, *resources*, *observables*). Like those three it is read by the parsing rule rather than by the modeling rule; it is declared for consistency with the file's own convention, so the dataset's JSON payload columns are described in one place.

#### Dashboards

##### KOI Content Extension - Alerts Dashboard

- **New: KOI Content Extension - Alerts Dashboard** — **53 widgets across 25 layout rows**, over a 30-day relative window, covering alert volume, item and MCP-server risk, device posture and marketplace breakdown. All widgets query **koi_koi_raw** directly and do not require the modeling rule.
- **Every counting or aggregating widget dedupes.** Because a row is a fetch and not an alert, each such widget counts `count_distinct(json_extract_scalar(metadata, "$.notification_event_id"))`, and each row list collapses to one row per notification with an *ingested_copies* column. The dedupe key is always that **inline** `json_extract_scalar`, never the promoted *koi_notification_id* column: parsing rules apply at ingest only, so the promoted column is null on every row ingested before this pack shipped and a dashboard built on it would render empty. For the same reason every other promoted column read here is `coalesce(<promoted>, <guarded raw read>)`.
- **New widget: *Koi Alert Ingest Duplication (rows vs distinct alerts)*** — the anti-regression exhibit. Two series per day, *ingested_rows* against *distinct_alerts*; the gap between them is re-sending, not alert volume. If a future edit reintroduces a raw `count()` anywhere on this dashboard, this is the widget where it becomes visible.
- Every marketplace facet is normalised to the Koi API vocabulary and renamed *marketplace_api* / *mcp_registry_api*, so a label an operator reads on a chart is a value they can paste into a `koi-*` command. Values with no API equivalent are labelled rather than silently mixed in: *built_in* / *side_loaded* render as *not-a-marketplace: <value>*, other unknowns as *unmapped: <value>*.
- **Koi MCP Servers by Registry** no longer hides MCP alerts whose registry is blank. It previously filtered them out; the registry is the empty string on **705 of 716** MCP resources (measured 21 July 2026), so the chart was showing 11. The blanks are now an explicit *(none)* slice, which is itself the security signal — most MCP servers come from no registry.
- Widgets that read the *item* resource cover the **extension** cohort only — 296 of 1,138 alert rows (26%), none since 2026-07-09, while current ingestion is effectively 100% *mcp_server*. Those widgets name the cohort in their title or header so an empty chart cannot be misread as "no risk". They are kept because the cohort can return.

#### Playbooks

##### KOI Ext - Alert Triage

- **New: KOI Ext - Alert Triage** — end-to-end triage of a KOI alert. Scores four signals, reaches a verdict, and either auto-closes the alert or hands off to response.
- **Duplicate-suppression gate (tasks 24-32) and two new inputs, *dedupe_alert_field* and *dedupe_lookback*.** Uncontrolled, this playbook runs once per delivery, so one real KOI alert becomes up to ~245 full investigations, device sweeps and war-room summaries — and, for a Malicious verdict, ~245 analyst approval gates. Set *dedupe_alert_field* to the machine name of the alert field carrying the KOI notification id and the playbook searches back over *dedupe_lookback* for an earlier alert with the same id, closing this one as a duplicate without investigating. The gate **fails open**: a missing notification id, an unconfigured field name or an unresolved `${alert.id}` skips it and the alert is triaged normally, so nothing here can close a first delivery. It is best-effort and not atomic, and it reacts *after* the alert exists — the alerts are still created and still count against the correlation rule's 5,000-hits-in-24-hours auto-disable ceiling. Suppressing at the correlation rule (Issue Suppression on *koi_notification_id*) is what prevents the alerts; this gate is defence in depth. Both options are documented in full in the playbook's own description.
- ***alert_type* branching removed.** The custom pack's version branched on "Removed from Marketplace" / "Publisher Compromised" / "Unvetted MCP Server" / "New Item". On live data *alert_type* has exactly one distinct value, *policy_violation*, on all 317 distinct alerts across the 1,048 alert rows measured 21 July 2026 — every one of those branches was dead. They were removed rather than shipped as dead code, and the verdict re-keyed onto signals that actually vary. *alert_type* is still shown in the triage summary; it is never tested.

#### Notes

- This pack extends the Palo Alto Networks **KOI** Marketplace content pack (pack *Koi*, integration id *KOI*, version **1.2.3** or later), which must be installed first. It ships no integration and adds no commands of its own — deliberately, to avoid overwriting the installed KOI integration.
- All playbook content is built strictly against the Marketplace pack's **13** commands. There is no `Koi.Device.*` context prefix; a device is addressable only as a filter on inventory (***koi-inventory-list*** with *device_id*).
- **Every live figure in this pack's documentation carries the date it was measured.** All figures above were measured on **21 July 2026** against tenant *api-ayman.xdr.eu*. **koi_koi_raw grows continuously**, so two queries taken minutes apart return different row totals — the 90-day dedupe measurement (1,048 rows → 317 notifications) and the 90-day cohort census (296 extension + 842 mcp_server = 1,138 rows) are separate snapshots and are quoted separately rather than reconciled into one total. Re-measure before quoting any of them as current.
- Minimum server version **8.4.0**.
